## ບົດຄັດຫຍໍ້

ບົດຄວາມນີ້ນຳສະເໜີ **Legalis-RS** ເຊິ່ງເປັນເຟຣມເວີກ Rust ສຳລັບການແຍກແລະຈັດໂຄງສ້າງເອກະສານກົດໝາຍພາສາທຳມະຊາດອອກເປັນ **ຕັກກະທີ່ກຳນົດໄດ້ແນ່ນອນ (Code)** ແລະ **ດຸນພິນິດຂອງສານ (Narrative)** ຢ່າງເຂັ້ມງວດ.

ລະບົບກົດໝາຍສະໄໝໃໝ່ປະກອບດ້ວຍສ່ວນປະສົມຂອງໂດເມນທີ່ສາມາດໃຊ້ຄອມພິວເຕີເຮັດວຽກອັດຕະໂນມັດໄດ້ (ຂໍ້ກຳນົດອາຍຸ, ເກນລາຍໄດ້, ການຄຳນວນກຳນົດເວລາ) ແລະໂດເມນທີ່ຕ້ອງການການຕີຄວາມໝາຍແລະການຕັດສິນຂອງມະນຸດ ("ເຫດຜົນອັນສົມຄວນ", "ສິນທຳອັນດີ"). ວິທີການກ່ອນໜ້ານີ້ໄດ້ປ່ອຍໃຫ້ຂອບເຂດນີ້ຄຸມເຄືອຫຼືພະຍາຍາມເຮັດໃຫ້ທຸກຢ່າງເປັນອັດຕະໂນມັດຫຼາຍເກີນໄປ.

Legalis-RS ນຳສະເໜີປະເພດຕັກກະສາມຄ່າ `LegalResult<T>` ໂດຍໃຊ້ລະບົບປະເພດຂອງ Rust ເພື່ອເຮັດໃຫ້ຂອບເຂດນີ້ຊັດເຈນໃນລະດັບປະເພດ. ສິ່ງນີ້ເຮັດໃຫ້ເກີດກະບວນທັດໃໝ່ສຳລັບການດີບັກກົດໝາຍ, ການຈຳລອງ, ແລະການຍ້າຍລະຫວ່າງປະເທດ, ໃນຂະນະທີ່ປ້ອງກັນ "ລະບອບຜະເດັດການອັລກໍຣິທຶມ" ໃນຍຸກ AI.

**ການປະກອບສ່ວນທາງເທັກນິກທີ່ສຳຄັນ**:
1. ພາສາສະເພາະໂດເມນທາງກົດໝາຍ (DSL) ແລະການນຳໃຊ້ຕົວແຍກວິເຄາະ
2. ການຢັ້ງຢືນຢ່າງເປັນທາງການດ້ວຍ Z3 SMT solver
3. ເຄື່ອງຈັກຈຳລອງແບບ ECS ສຳລັບການທຳນາຍຜົນກະທົບທາງສັງຄົມ
4. ການສ້າງສັນຍາສະຫຼາດສຳລັບແພລດຟອມບລັອກເຊນ 25+ ແພລດຟອມ
5. ການລວມ Linked Open Data (RDF/TTL) ສຳລັບເວັບເຊີງຄວາມໝາຍ
6. ການນຳໃຊ້ລະບົບກົດໝາຍສຳລັບ 4 ປະເທດພ້ອມການປັບຕົວແປທາງວັດທະນະທຳ (Soft ODA)

**ປັດຊະຍາຫຼັກ**: *"ບໍ່ແມ່ນທຸກຢ່າງທີ່ຄວນຈະຄຳນວນໄດ້"*

## 1. ບົດນຳ

### 1.1 ພື້ນຫຼັງ: ຄວາມສຳພັນລະຫວ່າງກົດໝາຍແລະການຄຳນວນ

ວິທະຍານິພົນທີ່ມີຊື່ສຽງຂອງ Lawrence Lessig "Code is Law" ຊີ້ໃຫ້ເຫັນວ່າສະຖາປັດຕະຍະກຳ (ໂຄ້ດ) ໃນໄຊເບີສະເປດມີອຳນາດຄວບຄຸມເທົ່າທຽມກັບກົດໝາຍ. ແນວໃດກໍຕາມ, Legalis-RS ກັບເຮັດສິ່ງນີ້ໂດຍໃຊ້ວິທີການ "**ກົດໝາຍກາຍເປັນໂຄ້ດ**".

ການເຂົ້າລະຫັດກົດໝາຍໃຫ້ຜົນປະໂຫຍດດັ່ງຕໍ່ໄປນີ້:

- **ຄວາມສາມາດໃນການຢັ້ງຢືນ**: ກວດຫາຄວາມຂັດແຍ່ງທາງຕັກກະໃນເວລາຄອມໄພລ໌
- **ການຈຳລອງ**: ທຳນາຍຜົນກະທົບທາງສັງຄົມກ່ອນການບັງຄັບໃຊ້
- **ຄວາມສາມາດໃນການເຮັດວຽກຮ່ວມກັນ**: ແປງແລະປຽບທຽບລະຫວ່າງລະບົບກົດໝາຍທີ່ແຕກຕ່າງກັນ
- **ຄວາມໂປ່ງໃສ**: ເສັ້ນທາງການກວດສອບທີ່ສົມບູນຂອງຂະບວນການຕັດສິນທາງກົດໝາຍ

ແນວໃດກໍຕາມ, ການເຮັດໃຫ້ກົດໝາຍທັງໝົດສາມາດຄຳນວນໄດ້ນັ້ນເປັນອັນຕະລາຍທັງໃນເຊີງປັດຊະຍາແລະໃນທາງປະຕິບັດ. ກົດໝາຍໂດຍເນື້ອແທ້ມີໂດເມນທີ່ຕ້ອງການ "ການຕັດສິນຂອງມະນຸດ", ແລະການເຮັດໃຫ້ເປັນອັດຕະໂນມັດທີ່ເພີກເຉີຍຕໍ່ສິ່ງນີ້ອາດນຳໄປສູ່ "ລະບອບຜະເດັດການ AI".

### 1.2 ການກຳນົດບັນຫາ: ສິ່ງທ້າທາຍຂອງການປະມວນຜົນກົດໝາຍໃນຍຸກ AI

ເທັກໂນໂລຢີກົດໝາຍສະໄໝໃໝ່ (LegalTech) ປະເຊີນກັບສິ່ງທ້າທາຍພື້ນຖານຫຼາຍຢ່າງ:

1. **ການຈັດການຄວາມຄຸມເຄືອ**: ຄຳສັບທາງກົດໝາຍຫຼາຍຄຳມີຄວາມຄຸມເຄືອໂດຍເຈດຕະນາ, ໂດຍຖືວ່າຕ້ອງຕີຄວາມໝາຍເປັນກໍລະນີໆ ໄປ
2. **ການຂຶ້ນກັບບໍລິບົດ**: ບົດບັນຍັດດຽວກັນອາດຖືກຕີຄວາມໝາຍແຕກຕ່າງກັນໄປຂຶ້ນກັບບໍລິບົດທາງສັງຄົມແລະວັດທະນະທຳ
3. **ການປ່ຽນແປງຕາມເວລາ**: ກົດໝາຍມີການແກ້ໄຂແລະຍົກເລີກ, ຕ້ອງມີການຈັດການຄວາມສອດຄ່ອງຂ້າມເວລາ
4. **ຄວາມແຕກຕ່າງລະຫວ່າງປະເທດ**: ລະບົບກົດໝາຍຂອງແຕ່ລະປະເທດແຕກຕ່າງກັນຕັ້ງແຕ່ຮາກຖານທາງປັດຊະຍາ

### 1.3 ຂໍ້ສະເໜີ: ການແຍກຄວາມສາມາດໃນການຄຳນວນແລະດຸນພິນິດຂອງສານ

ຫົວໃຈສຳຄັນຂອງ Legalis-RS ແມ່ນການນຳສະເໜີຕັກກະສາມຄ່າຜ່ານປະເພດ `LegalResult<T>`:

```rust
pub enum LegalResult<T> {
    /// [ໂດເມນທີ່ກຳນົດໄດ້] ຜົນລັບທາງກົດໝາຍທີ່ປະມວນຜົນໄດ້ອັດຕະໂນມັດ
    Deterministic(T),

    /// [ໂດເມນດຸນພິນິດ] ໂດເມນທີ່ຕ້ອງການການຕັດສິນຂອງມະນຸດ
    JudicialDiscretion {
        issue: String,           // ບັນຫາທີ່ເປັນຄຳຖາມ
        context_id: Uuid,        // ຂໍ້ມູນບໍລິບົດ
        narrative_hint: Option<String>, // ຄວາມເຫັນອ້າງອີງໂດຍ LLM
    },

    /// [ການລົ້ມເຫຼວທາງຕັກກະ] ຂໍ້ຜິດພາດໃນຕົວກົດໝາຍເອງ
    Void { reason: String },
}
```

ປະເພດນີ້ຮັບປະກັນວ່າຜົນລັບຂອງການປະມວນຜົນທາງກົດໝາຍຈະຖືກຈັດປະເພດເປັນໜຶ່ງໃນສາມໝວດໝູ່ສະເໝີ. ລະບົບຈະຢຸດການປະມວນຜົນເມື່ອຮອດ `JudicialDiscretion` ແລະມອບໝາຍການຕັດສິນໃຫ້ແກ່ມະນຸດ. ສິ່ງນີ້ກາຍເປັນ "ກຳແພງລະດັບປະເພດ" ຕໍ່ລະບອບຜະເດັດການ AI.

## 2. ວຽກທີ່ກ່ຽວຂ້ອງ

### 2.1 ປະຫວັດສາດຂອງກົດໝາຍເຊີງຄຳນວນ

ຄວາມສຳພັນລະຫວ່າງກົດໝາຍແລະຄອມພິວເຕີຍ້ອນກັບໄປຫາໂຄງການ LARC (Legal Analysis and Research Computer) ໃນຊຸມປີ 1950.

| ຍຸກ | ເທັກໂນໂລຢີ | ລັກສະນະສະເພາະ |
|-----|------------|-----------------|
| ຊຸມປີ 1950 | LARC | ລະບົບຄົ້ນຄືນຂໍ້ມູນກົດໝາຍທຳອິດ |
| ຊຸມປີ 1970 | ລະບົບຜູ້ຊ່ຽວຊານແບບ MYCIN | ການໃຫ້ເຫດຜົນຕາມກົດ |
| ຊຸມປີ 1980 | HYPO | ການໃຫ້ເຫດຜົນຕາມກໍລະນີ |
| ຊຸມປີ 1990 | ມາດຕະຖານ XML/SGML | ການຈັດໂຄງສ້າງເອກະສານກົດໝາຍ |
| ຊຸມປີ 2000 | Semantic Web | ການສະແດງຄວາມຮູ້ທາງກົດໝາຍຕາມອອນໂທໂລຢີ |
| ຊຸມປີ 2010 | Machine Learning | ໂມເດນການທຳນາຍທາງກົດໝາຍ |
| ຊຸມປີ 2020 | LLM + ການຢັ້ງຢືນຢ່າງເປັນທາງການ | ວິທີການແບບປະສົມປະສານ |

### 2.2 DSL ທາງກົດໝາຍທີ່ມີຢູ່

#### Catala (Inria, ຝຣັ່ງ)
- **ຄຸນສົມບັດ**: ການຂຽນໂປຣແກຣມແບບ literate, ຕາມຂອບເຂດ, ການກຳນົດປະເພດທີ່ເຂັ້ມງວດ
- **ຂໍ້ຈຳກັດ**: ບໍ່ມີການໝາຍໂດເມນດຸນພິນິດຢ່າງຊັດເຈນ

#### L4 (ສິງກະໂປ)
- **ຄຸນສົມບັດ**: ຕັກກະເຊີງບັນທັດຖານ (MUST/MAY/SHANT), ການໃຫ້ເຫດຜົນຕາມກົດ
- **ຂໍ້ຈຳກັດ**: ບໍ່ມີຟັງຊັນການຈຳລອງ

#### Stipula (ມະຫາວິທະຍາໄລໂບໂລຍາ, ອີຕາລີ)
- **ຄຸນສົມບັດ**: ມຸ່ງເນັ້ນສັນຍາສະຫຼາດ, ເຄື່ອງສະຖານະ, ໂມເດນຄູ່ສັນຍາ/ຊັບສິນ
- **ຂໍ້ຈຳກັດ**: ບໍ່ມີການຢັ້ງຢືນຢ່າງເປັນທາງການ

### 2.3 ຕຳແໜ່ງຂອງໂຄງການນີ້

Legalis-RS ຂະຫຍາຍວຽກຄົ້ນຄວ້າທີ່ມີຢູ່ໃນລັກສະນະດັ່ງຕໍ່ໄປນີ້:

1. **ການໝາຍດຸນພິນິດລະດັບປະເພດ**: ຕັກກະສາມຄ່າຜ່ານ `LegalResult<T>`
2. **ສະຖາປັດຕະຍະກຳແບບລວມ**: ທໍ່ Parse→Verify→Simulate→Output
3. **ຄວາມສາມາດໃນການເຮັດວຽກຮ່ວມກັນຫຼາຍຮູບແບບ**: ການແປງກັບ Catala/L4/Stipula/Akoma Ntoso
4. **ການອອກແບບຄວາມເປັນສາກົນ**: ການປັບຕົວແປທາງວັດທະນະທຳ (Soft ODA)
5. **ການລວມບລັອກເຊນ**: ການສ້າງສັນຍາສະຫຼາດສຳລັບ 25+ ແພລດຟອມ

## 3. ປັດຊະຍາແລະຫຼັກການອອກແບບ

### 3.1 "ການປົກຄອງໃນຮູບແບບໂຄ້ດ, ຄວາມຍຸຕິທຳໃນຮູບແບບການບັນຍາຍ"

ຄຳຂວັນຂອງ Legalis-RS ສະທ້ອນຄວາມແຕກຕ່າງທີ່ສຳຄັນລະຫວ່າງການປົກຄອງແລະຄວາມຍຸຕິທຳ:

- **ການປົກຄອງ**: ການໃຊ້ກົດ, ການປະຕິບັດຕາມຂັ້ນຕອນ, ການພິຈາລະນາຄຸນສົມບັດ → **ເຂົ້າລະຫັດໄດ້**
- **ຄວາມຍຸຕິທຳ**: ການບັນລຸຄວາມເປັນທຳ, ການຕີຄວາມໝາຍຕາມບໍລິບົດ, ການຕັດສິນຄຸນຄ່າ → **ເລົ່າເປັນການບັນຍາຍ**

### 3.2 ການອອກແບບຕັກກະສາມຄ່າ

ສາມຄ່າຂອງ `LegalResult<T>` ສອດຄ່ອງກັບແນວຄິດປັດຊະຍາຂອງກົດໝາຍດັ່ງຕໍ່ໄປນີ້:

| ປະເພດ | ແນວຄິດປັດຊະຍາຂອງກົດໝາຍ | ຕົວແທນການປະມວນຜົນ |
|------|---------------------------|------------------|
| `Deterministic(T)` | ກົດທີ່ໃຊ້ໄດ້ໂດຍອັດຕະໂນມັດ | ຄອມພິວເຕີ |
| `JudicialDiscretion` | ຫຼັກການທີ່ຕ້ອງການການຕີຄວາມໝາຍ | ມະນຸດ |
| `Void` | ຊ່ອງຫວ່າງທາງກົດໝາຍ/ຄວາມຂັດແຍ່ງ | ນັກນິຕິບັນຍັດ (ຕ້ອງແກ້ໄຂ) |

### 3.3 "ບໍ່ແມ່ນທຸກຢ່າງທີ່ຄວນຈະຄຳນວນໄດ້"

ຕໍ່ສິ່ງລໍ້ໃຈທີ່ຈະເຮັດໃຫ້ທຸກຢ່າງສາມາດຄຳນວນໄດ້, Legalis-RS ເວົ້າຢ່າງຊັດເຈນວ່າ "ບໍ່". ໂດເມນຕໍ່ໄປນີ້ໄດ້ຮັບການອອກແບບໂດຍເຈດຕະນາໃຫ້ບໍ່ສາມາດຄຳນວນໄດ້:

1. **ເຫດຜົນອັນສົມຄວນ**
2. **ຄວາມສະຫງົບຮຽບຮ້ອຍແລະສິນທຳອັນດີ**
3. **ຄວາມສຸຈະລິດ**
4. **ຄວາມສົມເຫດສົມຜົນ**

### 3.4 ການປ້ອງກັນລະບອບຜະເດັດການ AI

Legalis-RS ປ້ອງກັນລະບອບຜະເດັດການ AI ຜ່ານກົນໄກດັ່ງຕໍ່ໄປນີ້:

1. **ການຢຸດບັງຄັບໂດຍປະເພດ**: ຢຸດອັດຕະໂນມັດເມື່ອຮອດ `JudicialDiscretion`
2. **ເສັ້ນທາງການກວດສອບບັງຄັບ**: ການບັນທຶກຂະບວນການຕັດສິນທັງໝົດ
3. **ຄວາມສາມາດໃນການອະທິບາຍ**: ຜົນລັບທີ່ມີໂຄງສ້າງຂອງເຫດຜົນການຕັດສິນ
4. **ການຮັບປະກັນວົງຈອນມະນຸດ**: ມະນຸດຕັດສິນຂັ້ນສຸດທ້າຍໃນໂດເມນດຸນພິນິດສະເໝີ

### 4.2 ຊັ້ນແກນກາງ

**ນິຍາມປະເພດຫຼັກ**:

    VERSION 2
    EFFECTIVE_DATE 2024-01-01

    WHEN AGE >= 18 AND HAS citizen
    THEN GRANT "ສິດໃນການລົງຄະແນນສຽງ"

    DISCRETION "ການພິຈາລະນາຄວາມສາມາດທາງຈິດຕ້ອງມີການວິນິໄສຈາກແພດ"

## 5. ເທັກໂນໂລຢີຫຼັກ

### 5.1 DSL ທາງກົດໝາຍ

**ໂຄງສ້າງພື້ນຖານ**:
```
STATUTE <id>: "<title>" {
    [JURISDICTION "<jurisdiction>"]
    [VERSION <number>]
    [EFFECTIVE_DATE <date>]

    WHEN <condition>
    THEN <effect>

    [EXCEPTION WHEN <condition>]
    [DISCRETION "<description>"]
}
```

### 5.2 ປະເພດ LegalResult<T> ແລະຄ່າຄວາມຈິງບາງສ່ວນ

```rust
pub enum LegalResult<T> {
    Deterministic(T),
    JudicialDiscretion {
        issue: String,
        context_id: Uuid,
        narrative_hint: Option<String>,
    },
    Void { reason: String },
}
```

### 5.3 ການຢັ້ງຢືນຢ່າງເປັນທາງການດ້ວຍ Z3 SMT Solver

**ເປົ້າໝາຍການຢັ້ງຢືນ**:
1. ການອ້າງອີງແບບວົງກົມ
2. ກົດໝາຍທີ່ເຂົ້າເຖິງບໍ່ໄດ້
3. ຄວາມຂັດແຍ່ງທາງຕັກກະ
4. ຄວາມຂັດແຍ່ງກັບລັດຖະທຳມະນູນ

### 5.4 ເຄື່ອງຈັກຈຳລອງແບບ ECS

ເຄື່ອງຈັກຈຳລອງໃຊ້ຮູບແບບ Entity-Component-System (ECS):
- **Entity**: ຕົວແທນພົນລະເມືອງ
- **Component**: ຄຸນລັກສະນະ (ອາຍຸ, ລາຍໄດ້, ທີ່ຢູ່, ແລະອື່ນໆ)
- **System**: ຕັກກະການໃຊ້ກົດໝາຍ

### 5.5 ການສ້າງສັນຍາສະຫຼາດ

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract AdultVotingRights {
    struct Citizen {
        uint256 age;
        bool hasCitizenship;
    }

    function isEligible(Citizen memory citizen)
        public pure returns (bool)
    {
        return citizen.age >= 18 && citizen.hasCitizenship;
    }
}
```

## 7. ກໍລະນີສຶກສາ

### 7.1 ລະບົບພິຈາລະນາຄຸນສົມບັດສະຫວັດດີການ

**ຜົນລັບ**:
- **ການຕັດສິນທີ່ກຳນົດໄດ້**: 85% ຂອງກໍລະນີ
- **JudicialDiscretion**: 15% ຂອງກໍລະນີ

### 7.2 ການຈຳລອງປະມວນກົດໝາຍແພ່ງມາດຕາ 709 (ການລະເມີດ)

**ສະຖານະການທົດສອບ**:
1. ການລະເມີດໂດຍເຈດຕະນາທີ່ຊັດເຈນ → `Deterministic(Liable)`
2. ການລະເມີດໂດຍປະມາດ → `Deterministic(Liable)`
3. ກໍລະນີເທິງຂອບເຂດ → `JudicialDiscretion`

### 7.3 ການວິເຄາະປຽບທຽບກົດໝາຍການລະເມີດ 4 ປະເທດ

| ປະເທດ | ປະມວນກົດໝາຍ | ລັກສະນະສະເພາະ |
|---------|------|-----------------|
| ຍີ່ປຸ່ນ | ປະມວນກົດໝາຍແພ່ງ ມ. 709 | ບົດບັນຍັດທົ່ວໄປ |
| ເຢຍລະມັນ | BGB §823/§826 | ຜົນປະໂຫຍດທີ່ໄດ້ຮັບການຄຸ້ມຄອງທີ່ລະບຸໄວ້ |
| ຝຣັ່ງ | Code civil Art. 1240 | ການນາມມະທຳສູງສຸດ |
| ສະຫະລັດອາເມລິກາ | ກົດໝາຍຄະດີ | ຈຳແນກປະເພດ (Battery, ແລະອື່ນໆ) |

## 8. ຂໍ້ກຳນົດ API ແລະລາຍລະອຽດທາງເທັກນິກ

pub enum LegalResult<T> {

pub struct Statute {
    pub id: String,
    pub title: String,

    pub preconditions: Vec<Condition>,

## 11. ບົດສະຫຼຸບ

Legalis-RS ນຳສະເໜີວິທີການໃໝ່ໃນການເຂົ້າລະຫັດກົດໝາຍໂດຍເຮັດໃຫ້ "ຂອບເຂດລະຫວ່າງຄວາມສາມາດໃນການຄຳນວນແລະການຕັດສິນຂອງມະນຸດ" ຊັດເຈນໃນລະບົບປະເພດ.
